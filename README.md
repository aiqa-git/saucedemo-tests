## 🏗️ Структура проекта

* `tests/` — автотесты для интернет-магазина Saucedemo (выбор товаров, корзина, оформление заказа).
* `pages/` — реализация паттерна Page Object Model (POM) для разделения логики страниц и тестов.
* `code-reviewer/` — консольная утилита (`CLI-инструмент`) на Node.js, которая автоматически отправляет код тестов на аудит ИИ.

---

## 🤖 Модуль AI Code Reviewer

В проект интегрирован кастомный скрипт автоматического аудита кода, работающий на базе модели **Mistral AI**. Утилита проверяет файлы тестов по жестко заданному промпту, содержащему **6 ключевых критериев качества автотестов**:

1. **Надежность и стабильность тестов:** Защита от мигающих (`flaky`) тестов, правильная работа с асинхронностью и ожиданиями Playwright, уход от завязки на хрупкую верстку.
2. **Ассерты и проверяемость результата:** Использование правильных веб-проверок (`assertions`) со смысловыми сообщениями об ошибках, валидация именно тех состояний, которые задекларированы в тесте.
3. **Читаемость и намерение кода:** Понятная структура тест-кейса, логическое разбиение на шаги (`test.step`), стандарты именования и прозрачность бизнес-логики (Clear Intent).
4. **Хардкод и тестовые данные:** Изоляция тестовых данных, отсутствие жестко зашитых в код строк, конфигов или параметров авторизации.
5. **Повторяемость и DRY:** Минимизация дублирования кода (Don't Repeat Yourself), вынесение общих фикстур, базовых настроек и переиспользуемых методов.
6. **Архитектура тестов:** Соблюдение паттернов проектирования (в частности, Page Object Model), логичное разделение ответственности между классами страниц и самими тестами.

```javascript
await expect(page).toHaveURL(/inventory\.html$/); // Используем regex для частичного совпадения

// Тест 4: Заменить строгий toBe на toContainText
await expect(successMessage).toContainText('Thank you for your order!');
```

#### **4. Хардкод и тестовые данные**
**Статус:** [КРИТИЧНО]
**Проблема:**
- Логин/пароль (`standard_user`, `secret_sauce`) и тестовые данные (`Иван`, `Тестировщик`, `123456`) захардкожены.
- Это усложняет поддержку тестов и делает их менее гибкими.

**КАК ИСПРАВИТЬ:**
1. Вынести учетные данные в конфиг (например, `config/credentials.js`):
   ```javascript
   // config/credentials.js
   module.exports = {
     standardUser: {
       username: 'standard_user',
       password: 'secret_sauce',
     },
     testUser: {
       firstName: 'Иван',
       lastName: 'Тестировщик',
       zipCode: '123456',
     },
   };
   ```
2. Импортировать и использовать в тестах:
   ```javascript
   const { standardUser, testUser } = require('../config/credentials');

   // В beforeEach:
   await loginPage.login(standardUser.username, standardUser.password);

   // В тесте 4:
   await cartPage.fillCheckoutInformation(testUser.firstName, testUser.lastName, testUser.zipCode);
   ```

---

#### **5. Повторяемость и DRY**
**Статус:** [РЕКОМЕНДУЕТСЯ]
**Проблема:**
- В тесте `Тест 5` (негативный сценарий) логика логина дублируется с `beforeEach`.
- В тестах `Тест 2` и `Тест 3` используется `addFirstItemToCart()` и `removeFirstItemFromCart()`, но нет проверки, что товар действительно добавился/удалился (например, через проверку счетчика корзины).

**КАК ИСПРАВИТЬ:**
1. Для негативного сценария вынести логику логина в отдельный хелпер:
   ```javascript
   // helpers/auth.js
   async function loginWithWrongCredentials(page) {
     const loginPage = new LoginPage(page);
     await loginPage.navigate();
     await loginPage.login('standard_user', 'wrong_password');
   }

   module.exports = { loginWithWrongCredentials };
   ```
   Использовать в тесте:
   ```javascript
   const { loginWithWrongCredentials } = require('../helpers/auth');

   test('Тест 5: Ошибка авторизации при вводе неверного пароля', async ({ page }) => {
     await loginWithWrongCredentials(page);
     // ... остальной код
   });
   ```

2. Добавить проверки в тестах `Тест 2` и `Тест 3`:
   ```javascript
   // Тест 2: После добавления товара проверить, что он появился в корзине
   await inventoryPage.addFirstItemToCart();
   await expect(inventoryPage.getCartBadgeCount()).toBe('1');
   await expect(inventoryPage.getAddedItemsCount()).toBe(1); // Добавить метод в InventoryPage

   // Тест 3: После удаления товара проверить, что он исчез из корзины
   await inventoryPage.addFirstItemToCart();
   await inventoryPage.removeFirstItemFromCart();
   await expect(inventoryPage.getCartBadgeCount()).toBe('0');
   await expect(inventoryPage.getAddedItemsCount()).toBe(0);
   ```

## 🛠️ Инструкция по настройке и запуску проекта

Если вы хотите запустить проект локально, выполните следующие шаги:

1. Перейти в папку ИИ-ревьюера из корня проекта:
cd code-reviewer

2. Установить необходимые зависимости:
npm install

3. Создать в папке code-reviewer файл с именем .env и добавить туда личный API-ключ:
MISTRAL_API_KEY=ваш_реальный_ключ_api

4. Запустить ИИ-анализ файла автотестов:
node index.js ../tests/saucedemo.spec.js
